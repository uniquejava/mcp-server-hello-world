## UCF 分支做了什么
是的，当前就在 `ucf` 分支上。

**结论**
- 当前分支是 `ucf`，`git branch --show-current` 返回 `ucf`。
- 这个分支的核心改动是：把 Databricks Unity Catalog 里的函数自动发现出来，并动态注册成 MCP tools。

**它做了什么**
- 在启动时读取环境变量 `DATABRICKS_UC_FUNCTIONS_CATALOG`、`DATABRICKS_UC_FUNCTIONS_SCHEMA`、`DATABRICKS_UC_TOOL_PREFIX`，决定去哪个 `catalog.schema` 做发现，以及 tool 名称前缀。见 `<ROOT>/server/ucf/discovery.py:13`
- 通过 Databricks SDK 的 `workspace_client.functions.list()` 列出 schema 下的函数，再对每个函数调用 `functions.get()` 拉完整 metadata。见 `<ROOT>/server/ucf/discovery.py:27`
- 把每个 UC function 的 metadata 转成一个真正的 Python/FastMCP tool 函数，并注册到 MCP server。见 `<ROOT>/server/ucf/discovery.py:50`
- 在原来的 `load_tools()` 末尾接入这套自动注册逻辑，所以除了已有的 `health`、`get_current_user`，还会多出一批动态工具。见 `<ROOT>/server/tools.py:19` 和 `<ROOT>/server/tools.py:110`

**动态工具是怎么生成的**
- 会把 UC function 名称清洗成合法的 MCP tool 名，例如把非法字符替换成 `_`，数字开头时补 `ucf_`。见 `<ROOT>/server/ucf/executor.py:35`
- 会把常见 SQL 参数类型映射成 Python 类型，用于生成 tool 的参数签名；暂时主要支持 `BOOLEAN/FLOAT/DOUBLE/INT/LONG/SHORT/STRING`，其他类型先退化成 `str`。见 `<ROOT>/server/ucf/executor.py:24` 和 `<ROOT>/server/ucf/executor.py:46`
- 它不是手写一堆 wrapper，而是运行时拼接函数源码，再 `exec()` 出一个带真实参数签名的 tool function。见 `<ROOT>/server/ucf/executor.py:234`

**调用时怎么执行**
- 每个动态 tool 最终会走 `execute_function()`，把传入参数绑定成 SQL statement parameters。见 `<ROOT>/server/ucf/executor.py:135`
- 标量函数执行 SQL 形如 `SELECT catalog.schema.fn(:arg1, ...) AS result`；表函数执行 SQL 形如 `SELECT * FROM catalog.schema.fn(:arg1, ...)`。见 `<ROOT>/server/ucf/executor.py:123`
- 实际执行走 Databricks SQL Statement Execution API，需要 `DATABRICKS_WAREHOUSE_ID` 或调用时显式传 `warehouse_id`。见 `<ROOT>/server/ucf/executor.py:137`
- 返回值会尽量整理成自然结构：单值返回标量，一行多列返回 dict，多行返回 list。见 `<ROOT>/server/ucf/executor.py:108`

**整体流程**
- 应用启动 → `load_tools()` 注册静态工具 → 自动 discovery UC functions → 为每个函数生成 MCP tool → 客户端调用某个 tool → tool 内部执行对应 UC function。入口见 `<ROOT>/server/app.py:20`
